# 四大场景方案对比与面试总结

本文结合 Gomall 项目代码与业界常见做法，对以下四个场景做**现有方案总结**与**多种方案对比**，便于面试时清晰表述。

---

## 一、JWT + Casbin 实现细粒度权限控制

### 1.1 本项目现有方案

**认证（JWT）**

- **位置**：`app/auth/biz/middleware/jwt_middleware.go`
- **流程**：
  1. 除 `DeliverTokenReq`（登录/发 token）外，所有请求需带 Token。
  2. 使用 `jwt.Parse` + HS256 验证，密钥目前写死（`your-256-bit-secret`）。
  3. 从 `jwt.MapClaims` 解析 `user_id`、`user_role`，可选 `resource`、`action`，写入 `context`。
  4. 未传或无效 token 返回 `invalid token`。
- **发 Token**：`deliver_token_by_rpc.go` 用 `CustomClaims`（含 UserId、UserRole、Resource、Action、过期时间 24h）签发 JWT。

**授权（Casbin）**

- **位置**：`app/auth/biz/middleware/casbin_middleware.go`、`app/auth/conf/casbin/model.conf`、`app/auth/main.go`
- **模型**：RBAC，`model.conf` 中 `r = sub, obj, act`，`p = sub, obj, act`，`g = _, _`（角色继承），匹配规则 `m = g(r.sub, p.sub) && r.obj == p.obj && r.act == p.act`。
- **策略存储**：GORM Adapter，策略持久化在 DB（如 `casbin_rule` 表），启动时 `LoadPolicy()`，空则 `loadDefaultPolicies`（admin/seller/customer 对 product/order/cart 等权限）。
- **中间件逻辑**：
  1. 从 context 取 `user_role`、`resource`、`action`（无则默认 customer）。
  2. 先 `Enforce(userRole, resource, action)`。
  3. 若不通过且 `userID > 0`，再 `Enforce(userRole, resource, action, "user:"+userID)` 做资源级（如本人订单）校验。
- **链式顺序**：在 auth 服务中先 `JWTAuthMiddleware` 再 `CasbinMiddleware`，形成认证 → 授权。

**小结**：认证用 JWT 无状态、带角色与可选的资源/操作；授权用 Casbin RBAC + 数据库策略 + 可扩展的资源级（owner）校验。

---

### 1.2 常见方案对比

| 维度 | 本项目 | 方案 A：Session + 简单角色 | 方案 B：OAuth2/OIDC + ABAC | 方案 C：JWT + 自研权限表 |
|------|--------|---------------------------|----------------------------|--------------------------|
| **认证** | JWT（HS256，密钥写死） | Session（服务端存储） | OAuth2/OIDC（IdP 发 token） | JWT | 
| **授权** | Casbin RBAC + 可选 owner | 内存/DB 角色列表，代码里 if 判断 | ABAC（属性策略，如部门、级别） | 自建 permission 表，按 role 查权限再判断 |
| **细粒度** | 资源 + 操作，可加 owner | 多为接口级或菜单级 | 属性级，非常细 | 取决于表设计（可做到资源+操作） |
| **策略存储** | 数据库（GORM Adapter） | 配置或 DB | 策略引擎/策略文件 | 数据库 |
| **扩展性** | 加策略、加角色即可，支持动态 | 加角色要改代码或配置 | 改属性与策略即可 | 改表与业务代码 |
| **适用** | 中后台、多角色、需资源级控制 | 简单后台、角色少 | 企业 SSO、复杂策略 | 不想引入 Casbin 时 |

**面试可答要点**

- 我们采用 **JWT 做认证、Casbin 做授权**：JWT 解决“是谁”，Casbin 解决“能做什么”。
- 细粒度体现在：**角色 × 资源 × 操作**，必要时再叠加 **资源所有者**（如 user:123）。
- 策略存库便于运维动态配置，无需发版；RBAC 模型简单，易与现有角色体系对接。

### 1.3 应有设计（多版迭代）

当前缺失或可增强点：**resource/action 未从各业务请求自动注入**（依赖 token 或上游传）、**JWT 密钥写死**、**无统一权限管理 API**、**非 auth 服务未统一走权限链**。下面按版本递进设计。

| 版本 | 目标 | 设计要点 |
|------|------|----------|
| **V1：补齐基础** | 密钥与资源映射可配置；auth 链路闭环 | 1）JWT 密钥从配置/环境变量读取，支持 RS256 可选。2）在 auth 的 VerifyToken 或网关层，根据请求 path/method 映射到 `resource`、`action`（如 GET /order → order, read），写入 token 或 context。3）DeliverToken 时若未传 resource/action，按“默认资源”或登录角色给一个默认权限集。 |
| **V2：多服务统一鉴权** | 各 RPC 服务也能做细粒度校验 | 1）网关或 BFF 统一调 auth.VerifyToken，拿到 claims 后把 `user_id`、`user_role`、`resource`、`action` 通过 metadata 传给下游。2）或在 order/cart/product 等服务内引入轻量鉴权：从 metadata 取 identity，调 auth 的 CheckPermission RPC，或本地嵌入 Casbin + 定期同步策略。3）资源映射表：path/method → (resource, action)，可配置在 DB 或配置文件。 |
| **V3：动态策略与审计** | 运维可配置策略；可审计 | 1）提供权限管理 API：AddPolicy、RemovePolicy、AssignRole、GetUserRoles（本项目已有 policy_service，可暴露为 HTTP/RPC）。2）支持“资源所有者”动态策略：如 `p, customer, order, read, user:{id}` 由业务在创建订单时写入。3）关键操作打审计日志（谁在何时对哪资源做了何操作），便于安全与合规。 |

**迭代小结**：V1 解决配置与映射；V2 解决多服务一致鉴权；V3 解决运营态策略与审计，形成完整细粒度权限体系。

## 二、基于 Redis 分布式锁实现防止超买

### 2.1 本项目现有方案

**实际防超买手段：Redis + Lua 原子扣减（未用分布式锁）**

- **位置**：`app/checkout/biz/saga/checkout_saga.go` 中 `decreaseStockAction`。
- **流程**：
  1. 按购物车商品逐个处理，库存 key：`{productId}_stock`。
  2. 先用 `GET` 读库存，校验是否 ≥ 购买数量；不足直接返回错误。
  3. 扣减用 **Lua 脚本** 在 Redis 内原子执行：`GET` → 判断 ≥ 购买量 → `DECRBY`，否则返回 0。
  4. 扣减结果 0 表示库存不足，返回错误；成功则把扣减记录写入 `sagaCtx["stock_decreases"]` 供后续补偿使用。

**Redis 分布式锁在本项目中的位置**

- **位置**：`app/checkout/biz/dal/redis/init.go` 的 `NewRedisLock`（基于 `go-redislock`）。
- **现状**：**仅封装了创建锁的能力，当前 checkout 流程中并未使用**。防超买完全依赖上述 **Redis + Lua 原子扣减**，无“先加锁再扣减”的步骤。

**小结**：本项目防超买 = **单 key 的 Lua 原子扣减**；分布式锁已具备，可用于“多 key 或复杂校验”的扩展方案。

---

### 2.2 常见方案对比

| 方案 | 实现方式 | 优点 | 缺点 | 适用 |
|------|----------|------|------|------|
| **本项目：Lua 原子扣减** | GET + Lua（判断+DECRBY）单 key | 无锁、性能好、实现简单、原子性由 Redis 保证 | 多商品时需多次 Lua；库存 key 分散，无“整单”锁 | 单商品或按商品逐个扣减的秒杀/下单 |
| **Redis 分布式锁 + 扣减** | 锁 key（如 `lock:stock:{id}`）→ 查库存 → 扣减 → 解锁 | 可包住 DB 或复杂逻辑，易理解 | 锁竞争大时吞吐下降，需处理续期、死锁 | 扣减逻辑复杂或需与 DB 强一致时 |
| **DB 行锁 / 乐观锁** | SELECT FOR UPDATE 或 version 更新 | 与 DB 一致性强，适合库存落库 | 高并发下 DB 压力大、易阻塞 | 库存主要在 DB、并发不极端 |
| **预扣/占库存 + 异步** | 先 Redis 扣减，异步同步 DB 或 MQ 落库 | 读多写少、性能好 | 一致性与对账复杂 | 大促、秒杀，可接受最终一致 |
| **令牌桶/限流** | 用 Redis 做限流，再配合上面任一种 | 防止瞬时超卖、保护下游 | 不解决库存一致性问题，需叠加扣减方案 | 作为补充手段 |

**若用 Redis 锁防超买（可选增强）**

- 锁 key：如 `lock:stock:{productId}` 或 `lock:order:{userId}`（按业务选粒度）。
- 流程：`Lock()` → 查库存（Redis 或 DB）→ 扣减 → `Unlock()`，锁内逻辑尽量短。
- 注意：设置合理过期时间、用 Lua 或 SET NX 保证加锁原子性、考虑 Redlock 或多机部署时的可靠性（本项目当前为单 Redis）。

**面试可答要点**

- 我们**没有在扣减前加分布式锁**，而是用 **Redis Lua 脚本** 做“读-判断-扣”的原子操作，避免超卖且减少锁竞争。
- 项目里已提供 **Redis 锁封装**（`NewRedisLock`），若后续要做“整单锁”或“多 key 一起扣”可以在此基础上扩展。
- 对比：**Lua 适合单 key 高并发**；**分布式锁适合跨 key 或跨存储的复杂事务**。

### 2.3 应有设计（多版迭代）

当前缺失：**分布式锁未接入扣减流程**；若业务需要“整单原子扣减”或“扣减与 DB 双写一致”，可按下述版本递进引入锁，并与现有 Lua 组合使用。

| 版本 | 目标 | 设计要点 |
|------|------|----------|
| **V1：单商品锁 + Lua** | 高并发下仍以 Lua 为主，锁仅做可选保护 | 1）对**单个商品**扣减：在 `decreaseStockAction` 中，对每个 `productId` 先 `lockKey := "lock:stock:" + productId`，`Lock(lockKey)`（过期如 5s），再执行现有 Lua 扣减，最后 `Unlock`。2）锁内逻辑仅：Lua 扣减 + 写 sagaCtx；若 Lock 失败可快速失败或短时重试。3）保留“无锁仅 Lua”的开关，便于压测对比。 |
| **V2：整单锁 + 批量扣减** | 防止同一用户并发多笔下单导致总扣减超出库存 | 1）锁 key：`lock:order:{userId}` 或 `lock:checkout:{requestId}`，在整单 checkout 开始时加锁，五步 Saga 全部完成或补偿后再释放。2）锁内顺序：获取锁 → 拉取购物车 → 对每个商品 Lua 扣减（或先校验总库存再批量 Lua）→ 后续创建订单等。3）锁过期时间略大于“正常 checkout 耗时”（如 30s），并考虑续期（若用支持续期的库）。 |
| **V3：多机与高可用** | 单 Redis 故障或多机房部署时的可靠性 | 1）锁实现：采用 Redlock（多 Redis 实例多数通过才认为加锁成功），或使用现成 Redisson/go-redislock 的 Redlock 模式。2）降级策略：Redlock 不可用时，可降级为“仅 Lua 扣减、不加锁”，并告警。3）库存双写场景：若需 Redis 与 DB 双写一致，可在锁内做“Redis Lua 扣减 + DB 行锁更新库存”，锁保证两个操作的原子视图。 |

**迭代小结**：V1 锁与 Lua 并存、单 key 粒度；V2 整单锁防重复下单与超扣；V3 多机与双写一致性，按业务需求选择落地到哪一版。

## 三、基于 MQ 实现回滚库存

### 3.1 本项目现有方案：Saga 编排 + 同步补偿（非 MQ）

**回滚库存方式：Saga 补偿，非消息队列**

- **位置**：`app/checkout/biz/saga/checkout_saga.go`、`common/saga/coordinator.go`。
- **流程**：
  1. Checkout 按顺序执行：减库存 → 创建订单 → 清空购物车 → 支付 → 标记订单已支付。
  2. 任一步失败，**协调器** 对已成功的步骤**逆序同步执行补偿**（如 `increaseStockCompensate`）。
  3. **回滚库存** 在 `increaseStockCompensate` 中实现：从 `sagaCtx["stock_decreases"]` 取出各商品扣减量，对 Redis 做 `INCRBY` 恢复。
  4. 协调器与补偿均在 **同一请求链路内同步执行**，**没有通过 MQ 发“回滚库存”消息**。

**MQ 在项目中的角色**

- 文档/架构中提到的 MQ（如 NATS）主要用于 **OLTP→OLAP 数据同步**（如 `docs/02-OLTP-OLAP双架构设计.md`），**不用于** checkout 的库存回滚。

**小结**：本项目库存回滚 = **Saga 编排 + 同步补偿**，MQ 未参与回滚逻辑。

---

### 3.2 常见方案对比

| 方案 | 实现方式 | 优点 | 缺点 | 适用 |
|------|----------|------|------|------|
| **本项目：Saga 同步补偿** | 协调器内逆序调补偿接口（如直接 INCRBY） | 实现简单、无 MQ 依赖、回滚及时 | 补偿失败需自己重试/记录；长流程会拉长请求时间 | 步骤可控、服务稳定、希望实现简单 |
| **MQ 异步回滚** | 某步失败发“回滚库存”消息，消费者执行 INCRBY | 解耦、削峰、消费者可重试 | 最终一致、需幂等、要监控堆积与死信 | 高并发、希望请求快速返回 |
| **本地消息表 + 定时任务** | 写本地表 + 发 MQ，定时扫表补发/补偿 | 消息不丢、可对账 | 实现复杂、有延迟 | 对一致性要求高且可接受延迟 |
| **TCC** | Try(预扣) / Confirm(确认) / Cancel(回滚) | 各资源可独立回滚、控制力强 | 业务侵入大、要写 Confirm/Cancel | 强一致、可接受改造成本 |
| **事务消息（RocketMQ 等）** | 先发半消息，本地事务成功则 commit 消息 | 保证“发消息”与“本地事务”一致 | 依赖 MQ 能力、有使用成本 | 已有 RocketMQ、希望消息与事务绑定 |

**若引入 MQ 做回滚库存（可选设计）**

- 下单/支付失败或超时后，向 MQ 发“库存回滚”消息（含 orderId、productId、quantity 等）。
- 消费者幂等：按 orderId+productId 去重，再执行 Redis INCRBY。
- 监控：死信队列、重试次数、告警，避免消息堆积导致库存长期不准。

**面试可答要点**

- 我们采用 **Saga 编排 + 同步补偿** 做库存回滚，**没有用 MQ**；回滚逻辑在协调器内直接执行，简单可控。
- 若流量进一步增大、希望请求快速返回，可以改为：**失败时发 MQ，由消费者异步回滚库存**，并做好幂等与监控。
- 能说清 **Saga 与 MQ 回滚的取舍**：同步补偿简单及时，MQ 异步解耦可扩展。

### 3.3 应有设计（多版迭代）

当前缺失：**回滚完全同步**，无 MQ 参与；在请求量上来或希望“下单接口快速返回”时，可引入 MQ 做异步回滚，并逐步增强可靠性。

| 版本 | 目标 | 设计要点 |
|------|------|----------|
| **V1：失败发 MQ，消费者回滚** | 解耦、请求快速返回 | 1）Saga 某步失败时，协调器在执行完同步补偿后（或**代替**部分同步补偿），向 MQ 发“库存回滚”消息。消息体：`{ orderId, userId, items: [{ productId, quantity }], reason, sagaId }`。2）独立消费者：消费后按 item 对 Redis 做 `INCRBY`；若某条已回滚过则跳过（见 V2 幂等）。3）下单接口在“发完 MQ 或记录失败状态”后即可返回，不阻塞在补偿逻辑上；可选：先做同步补偿保证及时，再发 MQ 做对账/二次回滚。 |
| **V2：幂等 + 重试 + 死信** | 不重复回滚、失败可重试、异常可排查 | 1）**幂等**：消费者先查“回滚记录表”（如 `stock_rollback_log(order_id, product_id, saga_id)`），已存在则直接 ack 消息不执行；执行成功再插入。2）**重试**：MQ 配置重试次数（如 3 次）与退避；失败进入死信队列。3）**死信**：死信消费者告警 + 落库，人工或定时任务补偿；或死信再次投递到“重试 topic”限流重试。4）监控：消费延迟、堆积数、死信条数。 |
| **V3：本地消息表或事务消息** | 发 MQ 与业务“要么都成要么都败” | 1）**方案 A - 本地消息表**：Saga 失败时在同一 DB 事务内：写“待回滚”记录到本地表（含 orderId、items、sagaId、status=pending），再 commit；定时任务或 CDC 扫 pending 发 MQ，发送成功更新 status=sent；消费者同上，并支持对账（扫本地表与 Redis 是否一致）。2）**方案 B - 事务消息（RocketMQ）**：协调器发事务半消息 → 执行“写回滚意图到 DB”的本地事务 → 提交/回滚半消息；消费者逻辑同 V2。3）V3 保证“只要记录了要回滚，最终一定会被消费或对账”，避免消息丢失。 |

**迭代小结**：V1 先跑通“失败→发 MQ→消费回滚”；V2 补幂等与可观测性；V3 在需要更强一致性时上本地消息表或事务消息，形成可演进的回滚方案。

## 四、策略模式支持多种支付、回调和处理超时

### 4.1 本项目现有方案

**策略模式：多种支付方式抽象**

- **位置**：`app/payment/biz/service/paymentProcessor.go`、`charge.go`。
- **接口**：`PaymentProcessor` 含 `Validate()`、`Execute() (uuid.UUID, error)`、`Name()`。
- **实现**：`CreditCardProcessor` 使用 `go-credit-card` 校验卡号等，`Execute()` 目前为本地生成 UUID 模拟成功。
- **创建**：`CreatePaymentProcessor(method string, info *payment.CreditCardInfo)` 按 method 分支，目前仅 `"creditcard"`，其它返回 unsupported。
- **流程**：Charge 时创建 Processor → `Validate()` → `Execute()` → 写 `PaymentAmount`、`PaymentLog` → 返回 `TransactionId`。

**回调与超时**

- **回调**：代码中**没有**支付渠道异步通知（如支付宝/微信回调）的处理逻辑；当前为同步“扣款即成功”。
- **超时**：**没有**看到订单超时未支付自动取消、定时查单、补偿任务等实现；Saga 的 RetryPolicy 是步骤级重试，不是“支付超时关单”。

**小结**：**策略模式已用于支付方式扩展**；**回调和超时未在本项目实现**，可作为“可扩展点”在面试中说明。

---

### 4.2 常见方案对比

| 维度 | 本项目 | 常见增强方案 |
|------|--------|--------------|
| **多支付方式** | 策略模式 + 工厂（CreatePaymentProcessor），当前仅信用卡 | 策略 + 工厂；可扩展 AlipayProcessor、WechatProcessor 等 |
| **回调** | 无 | 单独回调接口，验签 → 更新订单/支付状态 → 幂等（按 out_trade_no 等）→ 可能发 MQ 解耦 |
| **超时** | 无 | 定时任务扫“待支付”订单，超时关单并回滚库存；或延迟消息（MQ）触发关单 |

**策略模式扩展（面试可答）**

- 新增支付方式：实现 `PaymentProcessor`，在工厂中加分支或配置映射即可，符合开闭原则。
- 可配合配置中心或 DB 配置“渠道开关”，实现动态启用某支付方式。

**回调设计（常见做法）**

- 渠道调用我方 HTTP 接口，带签名与订单号/流水号。
- 验签 → 查订单状态（幂等）→ 更新已支付 → 若与下单分离，可发 MQ 通知订单/库存等。

**超时处理（常见做法）**

- 方案 1：定时任务每分钟/每 5 分钟扫“待支付且创建时间 > 30 分钟”的订单，关单并触发 Saga 补偿或发“回滚库存”消息。
- 方案 2：下单时发延迟消息（如 30 分钟后），消费者检查仍未支付则关单并回滚。
- 方案 3：支付页轮询或 WebSocket，前端超时跳转并可选调用“取消订单”接口。

**面试可答要点**

- 我们用 **策略模式 + 工厂** 支持多种支付方式，目前实现了信用卡，扩展新渠道只需加实现类与工厂分支。
- **回调与超时** 在当前项目未实现，但可以说明：回调需验签与幂等；超时可用定时任务或延迟消息 + 关单与库存回滚。

### 4.3 应有设计（多版迭代）

当前缺失：**无支付渠道异步回调**、**无订单超时未支付关单与库存回滚**。下面按版本递进：先补齐回调和超时，再扩展多渠道与对账。

| 版本 | 目标 | 设计要点 |
|------|------|----------|
| **V1：回调接口 + 超时关单（定时）** | 支持渠道异步通知；超时订单自动关闭 | 1）**回调**：提供 HTTP 接口，如 `POST /payment/notify/{channel}`。按 channel 路由到不同 Processor 的 `VerifyNotify(req)`（验签）→ 解析订单号、流水号、状态 → 查本地支付记录做**幂等**（已处理则返回 200 不重复更新）→ 更新订单状态为已支付、写支付成功记录 → 可选发 MQ 通知订单/积分等。2）**超时关单**：定时任务（如每 1 分钟）扫「订单状态=待支付且创建时间早于（当前时间−30 分钟）」的订单，批量关单：更新订单为已取消、触发库存回滚（调用现有 Saga 补偿或发 MQ 回滚消息）。3）支付记录表增加 `notify_at`、`notify_status` 便于对账与排查。 |
| **V2：超时用延迟消息 + 多渠道扩展** | 关单更精确；支持支付宝/微信等 | 1）**延迟关单**：下单或“创建待支付订单”时，发一条延迟消息（如 30 分钟后），body 含 orderId。消费者到点检查订单是否仍为待支付，是则关单并回滚库存；已支付则忽略。这样关单时间更准确，不依赖扫表间隔。2）**多渠道**：在 `CreatePaymentProcessor` 中按 `method`（如 alipay、wechat、creditcard）返回不同实现；各实现实现同一接口：Validate、Execute、VerifyNotify（用于回调验签与解析）。3）配置化：支付方式与渠道开关放配置中心或 DB，便于运营开关渠道。 |
| **V3：对账与补偿** | 与渠道对账、补单与异常恢复 | 1）**主动查单**：对“已发起支付但长时间未收到回调”的订单，定时调用渠道查单接口，若已支付则补写本地状态并触发后续逻辑（与回调路径复用更新逻辑，保证幂等）。2）**对账任务**：每日/每小时拉取渠道账单与本地支付记录对账，发现不一致则告警或生成差错单，人工或自动补单。3）**补偿**：关单或退款后，若发现库存/订单状态异常，通过补偿任务或 MQ 触发回滚与状态同步。 |

**迭代小结**：V1 先有回调和定时超时关单；V2 精确关单 + 多渠道策略扩展；V3 对账与补单，形成可运维的支付闭环。

## 五、面试速记表

| 场景 | 本项目做法 | 可对比的常见方案 | 一句话总结 |
|------|------------|------------------|------------|
| **JWT+Casbin** | JWT 认证 + Casbin RBAC（DB 策略 + 资源级 owner） | Session+角色、OAuth2+ABAC、自建权限表 | 无状态认证 + 可配置的细粒度授权 |
| **防超买** | Redis + Lua 原子扣减（未用锁） | Redis 锁+扣减、DB 行锁、预扣+异步 | 单 key 高并发用 Lua 更合适；锁留给复杂场景 |
| **回滚库存** | Saga 同步补偿（直接 INCRBY） | MQ 异步回滚、TCC、事务消息 | 当前同步补偿简单；高并发可改为 MQ 异步 |
| **支付** | 策略模式多支付；无回调/超时 | 策略+工厂；回调验签幂等；定时/延迟消息关单 | 扩展用策略；生产要补回调和超时关单 |

## 六、缺失方案的迭代总览（应有设计一览）

下面四个表格汇总各场景「从现状到完整」的递进版本，便于排期与面试时说明演进路径。

| 场景 | 当前状态 | V1（补齐基础） | V2（增强） | V3（完善/高可用） |
|------|----------|----------------|------------|---------------------|
| **JWT+Casbin** | 有认证授权；密钥写死；resource/action 依赖上游 | 密钥可配置；path→(resource,action) 映射 | 多服务统一鉴权；metadata 传 identity；可选 CheckPermission RPC | 动态策略 API；资源 owner 策略；审计日志 |
| **防超买** | 仅 Lua 原子扣减；有锁能力未用 | 单商品加锁 + Lua；可配置是否用锁 | 整单锁；防同一用户并发多笔超扣 | Redlock/多机；Redis+DB 双写时锁内一致 |
| **回滚库存** | Saga 同步补偿；无 MQ | 失败发 MQ → 消费者 INCRBY；接口快速返回 | 消费者幂等表；重试与死信；监控 | 本地消息表或事务消息；对账与不丢消息 |
| **支付** | 策略模式；仅信用卡；无回调/超时 | 回调接口验签+幂等；定时任务超时关单 | 延迟消息关单；Alipay/Wechat 等策略；渠道配置化 | 主动查单；对账任务；补偿与差错处理 |

**使用建议**：实现时可按列推进（先所有场景的 V1，再统一做 V2），或按行推进（先把某一场景做到 V3，再复制到其他场景）；面试时能说清「当前在哪儿、下一版做什么、最终形态是什么」即可。
